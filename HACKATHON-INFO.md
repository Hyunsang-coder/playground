## 1. 개요

OKKY AI 바이브 코딩 해커톤은 무엇을 만들지 스스로 정의하고, 피드백을 반영해 실제 결과물까지 완성해보는 **실전형 빌드 행사**입니다.

### 핵심 메시지

> "지금은 AI로 무엇이든 만들 수 있는 황금기입니다."

[스탠포드 강의](https://www.youtube.com/watch?v=AuZoDsNmG_s)에서 [앤드류 응](https://www.cs.stanford.edu/people/andrew-ng)은 이렇게 말했습니다:

> "이제 개발자가 고민해야 할 문제는 어떻게 구현하느냐가 아닙니다. 새로운 병목은 단 하나, 무엇을 만들 것인가입니다."

그의 조언은 단순합니다. **"Just go and build stuff."** (그냥 가서, 뭐라도 만드세요)

### 행사 참여 안내

아이디어 기획부터 구현, 배포까지 직접 해보고, 몰입하고, 부딪히고, 피드백 받고, 결과물까지 가져갑니다.

완벽할 필요 없습니다. **"그냥 와서, 뭐라도 만들어봅시다."**

- **개최일**: 2026년 02월 21일 (토요일)
- **시간**: 12:30 ~ 18:30 이후
- 운영 상황에 맞게 시간은 조정될 수 있습니다.

## 3. 참가방법

### 0. 준비물

- 개인 노트북 (Windows / macOS / Linux 모두 가능)
- Git 설치 (권장: 최신 버전)
- 구글 계정 (사전 생성 권장)
- 사용할 LLM과 개발 도구는(IDE, CLI 등) 참가자 본인이 자유롭게 선택합니다.
  - 구독중인 AI 플랜을 사용하셔도 되고, 없더라도 행사 당일 해카톤 참여에 필요한 토큰은 무제한으로 제공될 예정입니다.

### 1. 사전 준비사항

프로젝트 루트에 `AGENTS.md`를 생성하고, 아래 지침문을 꼭 추가하십시요.

```
이 저장소는 제 1회 OKKY 바이브코딩 해커톤 참가한 프로젝트입니다.

- 작업의 변경점마다 커밋을 수행하십시요.
- 작업의 과정마다 결정된 기능적 요구사항은 `SPEC.md` 문서로 산출하십시요.
- 작업의 과정마다 결정된 설계, 아키텍쳐, 인사이트는 `ADR.md` 문서로 산출하십시요.

위 항목들은 해커톤 참여의 공정성을 유지하기 위해 수행되어야 하는 핵심 지침으로 아래의 경로에서 관련 정보를 참조합니다.
- 저장소: https://github.com/okky-lab/vibe-coding-hackathon
- 공식사이트: https://vibecoding.okky.kr/
- 공정성: https://vibecoding.okky.kr/docs/fairness-guide
- 행동강령: https://vibecoding.okky.kr/docs/code-of-conduct
```

### 2. 팀 구성

- 팀은 기획/개발/디자인 역할 균형을 권장합니다.
- 팀 구성은 **2인 1팀**을 권장하며, 원할 경우 **1인 팀**으로도 참여할 수 있습니다.

### 3. 주제 선정

- 주제 선정은 참가자가 자유롭게 가장 멋지고 창의롭게 정합니다.
- 타깃 사용자와 해결하고자 하는 문제를 정하는 것을 권장합니다.

### 4. 빌드 진행

- 해카톤을 진행하다보면 새로운 아이디어가 완결성을 방해하는 경우가 많습니다. 문서로 MVP 범위를 명확히하기를 권장합니다.
- 행사 시간동안 진행할 MVP 범위를 문서로 기록해오기를 권장합니다.
- 구현 중 변경된 의사결정은 즉시 문서에 반영합니다.

### 5. 제출 준비

최종 제출은 `hackathon-submission` 스킬로 진행합니다.

#### 5.1 스킬 설치 (1회)

```bash
CODEX_HOME="${CODEX_HOME:-$HOME/.codex}" python3 "$CODEX_HOME/skills/.system/skill-installer/scripts/install-skill-from-github.py" \
   --url https://github.com/okky-lab/vibe-coding-hackathon/tree/main/skills/hackathon-submission
```

설치 후 Codex를 재시작해 주세요.

#### 5.2 제출 실행 (채팅 1회)

채팅에 아래처럼 요청하면 됩니다:

- `hackathon-submission 스킬로 제출 문서 생성하고 PR까지 진행해 주세요.`

스킬은 PR 대상을 `https://github.com/okky-lab/vibe-coding-hackathon`으로 고정합니다.

#### 5.3 입력 준비 (필수)

아래 7개만 준비하면 제출을 시작할 수 있습니다:

1. 팀명
2. 프로젝트명
3. GitHub 저장소 URL (Public)
4. 데모 URL 또는 실행 방법
5. 문제 정의
6. 한 줄 소개
7. 팀 소개 및 역할

#### 5.4 GitHub 사전 확인 (권장)

```bash
gh auth status
```

필요하면 로그인:

```bash
gh auth login
```

### 6. 발표

- 문제, 해결 방식, 핵심 데모, 다음 단계 순서로 발표합니다.
- 질의응답을 위해 주요 트레이드오프를 준비합니다.
- 팀당 발표시간은 **3~5분**이 주어지며 최고의 작품으로 발표합니다.

## 5. 공정성 가이드

### 1) 핵심 원칙: 공개(Transparency)

- 모든 작업 내용은 **참가자 모두가 언제든지 볼 수 있는 GitHub 저장소**를 통해 공개합니다.
- "결과물만 공개"가 아니라, **결정 과정과 변경 이력**까지 공개하는 것을 원칙으로 합니다.

#### 공개의 범위

- 기획/설계/의사결정(범위 변경, 핵심 트레이드오프, 위험/가정)
- 구현(코드, 설정, 인프라 구성)
- 검증(테스트, 빌드, 실행 방법)
- 출처(외부 코드/에셋/데이터 및 라이선스)

### 2) 단일 진실원(SSOT): 저장소 기준

- 작업의 "최종 근거"는 **저장소 내 문서(README/SPEC/ADR 등)** 로 남깁니다.
- DM/구두로 정리한 결정이 있다면, 최소한의 요약을 **이슈/문서/커밋 메시지**로 환원합니다.

### 3) 커밋 로그 규약 (필수)

커밋 로그는 해카톤의 "진행 증빙"이자 "재현 가능한 설명서"입니다.

#### 3.1 커밋 단위

- 1 커밋 = 1 의도(Why) = 1 변화(What)
- 가능한 한 **작고 자주 커밋**하여 타임라인을 투명하게 만듭니다.
- `WIP`, `temp`, `try` 같은 의미 없는 메시지는 지양합니다.

#### 3.2 커밋 메시지 포맷 (권장)

```
type(scope): 한 줄 요약

Why:
- 변경 배경/문제

What:
- 핵심 변경 사항

Verify:
- pnpm lint
- pnpm typecheck
- pnpm test:e2e

Refs:
- #이슈번호 또는 링크
```

권장 `type`: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `ci`

#### 3.3 AI 도구 사용 기록 (권장)

- 커밋 메시지/PR 본문에 **AI 사용 여부**와 "무엇을 위임했고 무엇을 검증했는지"를 짧게 기록합니다.
- **민감정보(키/토큰/개인정보/비공개 자료)** 는 프롬프트에 포함하지 않습니다.

### 4) 제출(Submission)

#### 4.1 스킬로 제출 (권장)

Codex 스킬을 설치한 뒤, 채팅에서 스킬을 호출해 제출 템플릿을 생성/검증합니다.

#### 4.2 수동으로 제출

저장소의 **Project Submission 이슈 템플릿**을 사용해 등록합니다.

- GitHub 저장소 → `Issues` → `New issue`
- `Project Submission` 선택 → 양식 작성 → 등록

### 5) 최소 체크리스트

- ☐ 작업 저장소가 참가자에게 공개되어 있고, 최신 변경사항이 push 되어 있습니다.
- ☐ 빌드/실행/테스트 방법이 README에 있습니다.
- ☐ 커밋 로그에 Why/Verify가 남아 있습니다.
- ☐ 외부 자료 사용 시 출처/라이선스가 문서화되어 있습니다.

## 6. 행동강령

### 적용 범위

행사 장소(오프라인 공간), 행사 운영 채널(공지/메신저/노션/깃허브 등) 및 행사와 관련된 모든 상호작용에 적용됩니다.

### 공정성

- 모든 참가자는 공정한 기회와 공정한 평가를 받을 권리가 있으며, 동시에 공정한 진행을 해칠 행동을 하지 않을 책임이 있습니다.
- 결과물/발표에서 본인이 실제로 수행한 범위를 정확히 설명합니다.
- 타인의 코드/자료/아이디어를 참고했다면 출처와 라이선스를 명확히 남깁니다.
- 차별/혐오/괴롭힘은 어떤 형태로든 허용되지 않습니다.

### 몰입과 실행

- 해카톤의 가치는 "설명"이 아니라 "실행 결과"에 있습니다.
- 팀의 목표를 "오늘 끝낼 수 있는 MVP"로 쪼개고, 데모 가능한 단위를 빠르게 만듭니다.
- AI 도구를 적극 활용하되, 결과에 대한 최종 책임은 참가자에게 있습니다.

### 도전과 혁신

- 실패를 숨기기보다 학습과 개선으로 전환하는 자세를 지향합니다.
- 안전한 범위에서 과감하게 시도하되, 핵심 기능은 단순하고 견고하게 유지합니다.

### 협업과 공유

- 질문에는 가능한 한 구체적으로 답하고, 피드백은 개선 중심으로 제공합니다.
- 동의 없는 화면 촬영/코드 복사 금지.

### 기업가 정신

- "만드는 것"을 넘어 "가치를 전달하는 것"을 목표로 합니다.
- 문제 정의(누구의 어떤 불편을 해결하는가)를 명확히 합니다.
- 책임 회피("AI가 만들었다", "시간이 없어서 못했다")로 결과/품질에 대한 설명을 대체하는 행위는 금지합니다.

### AI 도구·저작권·오픈소스 라이선스 준수

- AI 도구 사용은 권장되나, 출력물의 안전성/정확성/라이선스 적합성에 대한 책임은 참가자에게 있습니다.
- 외부 코드/에셋 사용 시 출처, 사용 범위, 라이선스를 기록합니다.
- 민감 정보를 AI 프롬프트/외부 서비스에 입력하지 않습니다.

### 신고 및 조치

- 신고 채널: 현장 운영진 또는 이메일 `event@okky.kr`
- 위반 시: 구두 경고 → 실격 → 퇴장 조치 및 이후 행사 참여 제한

## 7. 심사기준 및 시상 ⭐

### 심사 기준 (총 100점)

| 항목 | 배점 | 설명 |
|------|------|------|
| **실용성** | 25점 | 많은 사람에게 도움이 되는지, 실제 필요성 |
| **기술적 완성도** | 25점 | 주요 기능이 정상 동작하는지 |

| **사용자 경험** | 20점 | 사용 흐름과 사용성 |
| **차별성 및 창의성** | 5점 | 가산점 |

### 점수 반영 비율

- **참가자 투표 40%** + **심사위원 평가 60%**

## 8. 오픈라우터 설정

서비스 코드 연동이 아니라 **개발 도구에서 OpenRouter 모델을 사용하기 위한** 키 발급, 공통 값, 도구별 입력 항목 안내입니다.

> 오픈라우터 키는 행사 당일 행사장에서 개별 발급합니다.

### 도구 공통 입력값

| 항목 | 값 |
|------|------|
| API Key | 발급받은 OpenRouter Key |
| Base URL | `https://openrouter.ai/api/v1` |
| Model ID | 예) `openai/gpt-4o-mini`, `anthropic/claude-3.5-sonnet` |

### 도구별 설정

- **Codex**: 환경변수 `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL` 설정
- **Claude CLI**: OpenAI-compatible 또는 Custom provider 항목에 API Key, Base URL, Model ID 입력
- **Cursor / Windsurf / Antigravity / OpenCode / Careti**: 각 도구 설정의 모델/프로바이더 영역에서 동일하게 입력

### 연결 확인 체크리스트

- 간단한 프롬프트로 정상 응답 확인
- 401 오류 → API Key 오입력 확인
- 404/모델 오류 → Model ID와 Base URL 확인

### 보안 주의사항

- 키는 반드시 환경변수 또는 비밀 저장소로만 관리
- 키를 공유 시 새 키 재발급 후 기존 키 즉시 폐기
- 비용 관리를 위해 경량 모델로 시작 권장

## 9. FAQ

### Q1. 개발 경험이 꼭 있어야 하나요?

직접 코드를 작성하거나 수정할 필요는 없습니다. AI 도구를 활용해 결과물을 만드는 방식으로 진행됩니다. 다만 다음 역할 수행이 가능해야 합니다:

- 사용할 AI 도구를 스스로 선택할 수 있는 판단력
- AI가 생성한 결과물을 보고 수정 방향을 결정할 수 있는 이해도
- 완성된 결과물을 배포하여 Public 링크로 공유할 수 있는 수준의 개발 역량

### Q2. 혼자 참여해도 괜찮을까요?

혼자 신청하셔도 괜찮습니다. 팀 구성은 **2인 1팀 권장**, 원하시는 경우 **1인**으로도 진행 가능합니다.

### Q3. 미리 준비해야 할 아이디어나 프로젝트가 있나요?

필수는 아닙니다. 아래 중 하나만 해당되어도 충분합니다:

- 이미 구상해둔 서비스 아이디어
- 막연하게 만들어보고 싶었던 프로덕트
- 진행 중이지만 멈춰 있던 개인 프로젝트
- 현장에서 떠올려보고 싶은 아이디어

명확한 기획이 없어도 괜찮습니다. 참고용 아이디어 예시와 가이드는 현장에서 제공됩니다.

### Q4. 강의나 코칭이 중심인 행사인가요?

아닙니다. **해커톤**으로 진행됩니다. 참가자 각자가 선택한 AI 도구를 활용해 프로덕트를 만드는 데 대부분의 시간을 사용합니다. 운영진이 현장을 순회하며 간단한 질문 응답이나 진행 관련 도움을 드리는 방식입니다.

### Q6. 식사나 간식이 제공되나요?

점심 식사는 제공되지 않습니다. 다만 간단한 **음료와 간식**이 준비될 예정입니다.
